# 사용자 경험 기준

## 판단 우선순위

기능 구현 중 선택지가 생기면 아래 순서로 판단한다.

```text
정확성
  -> 사용자 경험
  -> 보안과 개인정보 보호
  -> 데이터 무결성
  -> 타입 안정성
  -> 유지보수성
  -> 성능
  -> 코드량
```

## 설명

- 정확하지 않은 기능은 사용자 경험도 좋을 수 없다.
- 사용자가 직접 마주하는 화면과 흐름에서는 사용자 경험을 우선한다.
- 조과 기록은 개인 데이터이므로 RLS, 인증, 이미지 접근 권한을 가볍게 보지 않는다.
- 통계 기능을 위해 물때, 날씨, 위치, 어종, 마릿수 데이터를 구조화해서 저장한다.
- 타입 에러를 `any`나 무리한 type assertion으로 덮지 않는다.
- 성능 최적화는 실제 병목이나 예상되는 쿼리 패턴이 있을 때 적용한다.
- 코드가 조금 길어져도 데이터 흐름이 명확한 쪽을 선택한다.

## 어장관리 UX 기준

- 라이트모드와 다크모드를 모두 지원한다.
- 사용자가 테마를 직접 선택하지 않았다면 시간대 기준으로 라이트/다크모드를 자동 적용한다.
- 어두운 환경에서 눈부심이 적어야 한다.
- 야외에서 빠르게 조작할 수 있도록 터치 영역과 대비를 충분히 확보한다.
- 조과 등록은 사용자가 낚시 직후 빠르게 입력할 수 있어야 한다.
- 필수값과 선택값을 명확히 구분한다.
- 이미지 업로드, 위치 선택, AI 어종 판별처럼 시간이 걸리는 작업에는 loading 상태를 보여준다.
- 실패 시 사용자가 다시 시도할 수 있는 메시지와 행동을 제공한다.
- AI 어종 판별 결과는 자동 확정하지 않고 사용자가 확인 후 수정할 수 있게 한다.
- 폼 입력 도중 화면 이동이나 실패로 사용자의 입력이 쉽게 사라지지 않게 한다.
- 빈 목록, 첫 기록 작성 전 상태, 이미지 없음 상태를 방치하지 않는다.
- 도감은 포켓몬 도감처럼 "채워가는 재미"가 느껴져야 한다.
- 잡은 어종과 아직 잡지 못한 어종의 상태 차이가 명확해야 한다.
- 도감 일러스트는 조과 사진과 구분되어야 한다.

## 테마 기준

1차 MVP는 라이트모드와 다크모드를 모두 지원한다.

테마 적용 우선순위:

```text
사용자가 직접 선택한 테마
  -> 시간대 기반 자동 테마
  -> 시스템 테마
  -> 기본 fallback 테마
```

사용자 선택이 없을 때의 기본 동작:

- 낮 시간대에는 라이트모드를 적용한다.
- 밤 시간대에는 다크모드를 적용한다.
- 정확한 시간대 기준은 구현 시 결정한다. 예: 18:00~06:00은 다크모드
- 사용자가 직접 라이트/다크를 선택하면 시간대 자동 전환보다 사용자 선택을 우선한다.

원칙:

- 다크모드 배경은 완전한 검정만 고집하지 않고, 눈부심이 적은 어두운 색을 사용한다.
- 라이트모드는 야외 주간 환경에서도 읽기 쉬운 대비를 가진다.
- 본문 텍스트와 주요 버튼은 라이트/다크 모두 충분한 대비를 가진다.
- 경고, 오류, 성공, 선택 상태는 색상만으로 구분하지 않고 텍스트나 아이콘 상태도 함께 제공한다.
- 지도, 사진, 도감 일러스트처럼 밝은 콘텐츠가 있는 화면에서는 주변 UI가 과하게 눈부시지 않게 한다.
- 밤낚시 중 한 손 조작을 고려해 주요 CTA는 누르기 쉬운 위치와 크기를 가진다.
- 테마 전환으로 layout이 흔들리면 안 된다.

## UI 변경 시 확인할 것

- 기대한 경로로 화면이 열린다.
- header가 중복으로 보이지 않는다.
- `router.push`, `router.replace`, `router.back` 선택이 의도와 맞다.
- Android safe area, keyboard, local server URL 이슈를 고려했다.
- list에 `undefined` 또는 잘못된 item이 들어가지 않는다.
