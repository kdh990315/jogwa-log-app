# Codex 작업 방식

## 기본 작업 흐름

Codex는 이 저장소 안에서 시니어 개발자처럼 작업해야 한다.

```text
요청 이해
  -> 관련 파일 먼저 읽기
  -> 가장 작고 안전한 변경 범위 찾기
  -> 기존 패턴을 따라 구현
  -> 관련 검증 명령 실행
  -> diff 자체 검토
  -> 변경 파일, 검증 결과, 남은 리스크 요약
```

## 작업 규칙

- 새 아키텍처를 먼저 만들지 말고 기존 코드를 먼저 읽는다.
- 변경 범위는 사용자 요청과 직접 관련된 파일로 제한한다.
- 새 추상화보다 기존 프로젝트 패턴을 우선한다.
- 기능 구현, 리팩터링, 포맷팅 변경, 의존성 업그레이드를 한 번에 섞지 않는다.
- 사용자가 만든 기존 변경사항을 되돌리지 않는다.
- 백엔드 동작이나 API 응답 형태가 확실하지 않으면 가정이라고 표시하고 방어적으로 작성한다.
- 명령을 실행하지 못했다면 이유와 사용자가 실행해야 할 명령을 남긴다.
- 잘못 추정하면 재작업이 큰 애매한 요청은 편집 전에 짧게 한 가지 질문을 한다.
- 단순 구현 요청은 계획만 말하지 말고 직접 구현한다.
- 더 나은 방법, 새 라이브러리, 설정 변경, 외부 연동 방식을 제안할 때는 최신 공식 문서를 먼저 확인한다.
- 공식 문서가 확인되지 않은 내용은 추정이라고 표시하고, 가능하면 공식 문서 확인이 필요한 항목으로 남긴다.

## 검증 규칙

코드를 수정한 뒤에는 가능한 한 lint와 TypeScript 타입체크를 모두 실행한다.

```bash
npm run lint
npx --no-install tsc --noEmit
```

규칙:

- `npm run lint`는 lint 검증이다.
- `npx --no-install tsc --noEmit`는 TypeScript 타입체크다.
- TypeScript 에러는 기능 완료 전에 해결한다.
- lint 에러는 기능 완료 전에 해결한다.
- 타입체크를 생략하지 않는다. 실행하지 못했다면 최종 답변에 이유를 명확히 적는다.
- 검증을 실행하지 못했으면 최종 답변에 이유를 명확히 적는다.
- 테스트 스크립트가 추가되면 관련 기능 변경 시 해당 테스트도 실행한다.
- Supabase schema나 RLS를 바꿨다면 migration SQL, RLS policy, 인덱스를 함께 검토한다.
- 인증, Storage, Edge Function 변경은 보안 리스크를 최종 답변에 따로 언급한다.
- Supabase, Expo, React Native, Kakao, Gemini, Apple/Google 로그인처럼 외부 서비스나 라이브러리 사용법을 제안하거나 바꿀 때는 최신 공식 문서를 기준으로 판단한다.

## 먼저 계획해야 하는 경우

아래 경우에는 파일을 수정하기 전에 먼저 계획을 세운다.

- 여러 화면, 공통 hook, API 계약, 라우팅 구조를 함께 건드리는 작업
- 원인이 불명확한 버그 수정
- 새 라이브러리나 새 구조가 필요한 작업
- 사용자가 리뷰, 마이그레이션, 리디자인, 큰 기능을 요청한 경우

계획에는 아래 내용을 포함한다.

- 바뀔 가능성이 높은 파일
- 영향을 받는 데이터 흐름 또는 라우트 흐름
- 실행할 검증 명령
- 주요 리스크 또는 가정

계획이 명확해진 뒤에는 위험한 결정에 대한 사용자 승인이 필요한 경우가 아니라면 바로 구현한다.

## 완료 기준

작업은 아래 조건을 만족해야 완료로 본다.

- 요청한 동작이 구현되어 있다.
- 타입이 실제 runtime 데이터와 맞는다.
- 필요한 경우 loading, empty, error, invalid-param 상태를 처리했다.
- cache invalidate 또는 optimistic update 범위가 맞다.
- navigation 동작이 route 구조와 맞다.
- lint 검증을 실행했다. 보통 `npm run lint`
- TypeScript 타입체크를 실행했다. 보통 `npx --no-install tsc --noEmit`
- 최종 답변에 변경 파일, 검증 결과, 남은 리스크를 요약했다.

## Codex 고급 활용

작업이 크거나 애매하면 Plan mode를 사용하거나 먼저 계획을 요청한다.

parallel/subagent 작업은 사용자가 명시적으로 요청했고, 일이 서로 독립적일 때만 사용한다.

예:

- 한 agent는 navigation과 route 흐름을 조사한다.
- 한 agent는 API와 query cache 흐름을 조사한다.
- 한 agent는 test 또는 regression 가능성을 검토한다.

같은 파일을 여러 Codex thread가 동시에 수정하지 않게 한다. 병렬 작업이 필요하면 별도 worktree를 사용한다.

MCP는 repo 밖의 최신 정보가 필요할 때만 사용한다.

예:

- GitHub issue 또는 PR
- 디자인 파일
- 자주 바뀌는 API 문서
- 에러 추적 도구 또는 로그

반복되는 작업은 skill로 만든다.

예:

- PR 리뷰 체크리스트
- 릴리즈 노트 작성
- API 마이그레이션 계획
- 표준 버그 triage
- 디자인을 코드로 옮기는 절차
